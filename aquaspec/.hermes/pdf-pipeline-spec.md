# PDF Pipeline — Full Specification
**For:** AquaSpec Coder  
**Date:** 2026-07-01  
**ADR:** ADR-007 (see `docs/ADRs/ADR-007-pdf-pipeline.md`)

---

## Goal
Enable sales personnel to generate a branded A4 PDF proposal from the sizing engine results. User clicks "Download PDF" → sees a branded full-page preview → clicks Download → receives `LotusOzone_Proposal_{HatcheryName}_{Date}.pdf`.

---

## Context
The `ProposalPreview` modal currently has a placeholder `console.log("not implemented")` on the Download button. The sizing engine produces full `HatcheryRecommendation` data. The Lotus Ozone logo is at `public/LotusOzoneLogo.png` (700×301px). Primary brand color is `#1F5DE1`. The user wants a dedicated HTML template, Playwright for PDF generation, and the budgetary estimate toggle built now.

---

## Architecture Overview

```
ProposalPreview modal  ──[Download PDF]──→  /proposal/preview (preview page)
                                                    │
                                                    ├── Reads Zustand store
                                                    ├── POST /api/proposal/preview → returns HTML
                                                    └── [Download PDF button]
                                                         │
                                                         └── POST /api/proposal/pdf
                                                              ├── Reads template
                                                              ├── Generates body HTML
                                                              ├── Playwright → PDF
                                                              └── Returns PDF blob
```

### 1c. `src/lib/proposal-html.ts`

Pure TypeScript module. Zero React imports. Exports functions that generate HTML strings from `HatcheryRecommendation` data.

**Functions to export:**

```typescript
import type { HatcheryRecommendation, SystemRecommendation } from "./sizing-engine/types";

/** Generate the systems section HTML */
export function renderSystemsHtml(recommendation: HatcheryRecommendation): string;

/** Generate the aggregate summary HTML (returns empty string if mode is aggregate) */
export function renderAggregateHtml(recommendation: HatcheryRecommendation): string;

/** Generate the budgetary estimate HTML section */
export function renderBudgetaryHtml(pricing: Record<string, string>): string;

/** Replace all tokens in the template */
export function applyTemplate(template: string, tokens: Record<string, string>): string;
```

**`renderSystemsHtml` logic:**
For each system in `recommendation.systems`, generate:
- System name heading
- Input parameters table (water source, quality, salinity, volume, turnovers, op. hours, pathogen, species, system type)
- Engineering results table (flow rate m³/hr, ozone demand g/hr → model, UV model + parallel units, oxygen model, total oxygen m³/hr)
- In multi-system mode, add `class="page-break"` before each system after the first

**`renderAggregateHtml` logic:**
If `mode === "aggregate"`: return `""`.
If `mode === "multi_system"` and `aggregateSummary` exists, render a table with total flow, total ozone, total oxygen, combined equipment list.

**`renderBudgetaryHtml` logic:**
Read pricing strings from the `pricing` record. Render a div with class "budgetary" containing each category's text block and the disclaimer.

**`applyTemplate` logic:**
`String.replace()` each `{{TOKEN}}` with its value. Simple, no templating engine.

## Slice 2: API Routes

### 2a. `src/app/api/proposal/preview/route.ts`

**Method:** POST  
**Request body:** `{ recommendation: HatcheryRecommendation, includeBudgetary: boolean }`  
**Response:** `text/html` — the fully rendered proposal HTML

**Logic:**
1. Read `src/templates/proposal.html` from filesystem (`fs.readFileSync` with path relative to `process.cwd()`)
2. Read `src/data/pricing.json`
3. Import and call `renderSystemsHtml()`, `renderAggregateHtml()`, `renderBudgetaryHtml()` from `src/lib/proposal-html.ts`
4. Build token map:
   - `PRIMARY_COLOR`: `#1F5DE1`
   - `LOGO_URL`: resolve `/LotusOzoneLogo.png` to absolute URL using `req.nextUrl.origin` (from NextRequest)
   - `HATCHERY_NAME`, `DATE`, `MODE_LABEL`: from recommendation
   - `SYSTEMS_HTML`, `AGGREGATE_HTML`: from render functions
   - `BUDGETARY_HTML`: from `renderBudgetaryHtml` if `includeBudgetary` is true, else `""`
   - `FOOTER_TEXT`: `"Generated by AquaSpec — Lotus Ozone Tech Pvt. Ltd."`
5. Apply template
6. Return `new NextResponse(html, { headers: { "Content-Type": "text/html" } })`

**Error handling:** Return 500 with `{ error: string }` on filesystem errors.

### 2b. `src/app/api/proposal/pdf/route.ts`

**Method:** POST  
**Request body:** `{ recommendation: HatcheryRecommendation, includeBudgetary: boolean }`  
**Response:** `application/pdf` — the PDF file as a blob download

**Logic:**
1. Reuse the SAME HTML rendering logic as the preview route (steps 1-5 above). **Do not duplicate** — refactor the shared HTML generation into `src/lib/proposal-html.ts` as `renderProposalHtml(recommendation, includeBudgetary, logoUrl): string`.
2. Launch Playwright Chromium: `const browser = await chromium.launch()`
3. Open a new page, set content to the HTML: `await page.setContent(html, { waitUntil: "networkidle" })`
4. Generate PDF: `await page.pdf({ format: "A4", printBackground: true, margin: { top: "20mm", bottom: "20mm", left: "20mm", right: "20mm" } })`
5. Close browser: `await browser.close()`
6. Return: `new NextResponse(pdfBuffer, { headers: { "Content-Type": "application/pdf", "Content-Disposition": `attachment; filename="LotusOzone_Proposal_${hatcheryName}_${date}.pdf"` } })`

**Error handling:**
- **15 second timeout:** Wrap Playwright calls in `Promise.race([pdfPromise, timeout(15000)])`. On timeout, return 504 with `{ error: "PDF generation timed out. Please try again." }`
- **Chromium unavailable:** Catch launch errors. Retry ONCE. If retry also fails, return 503 with `{ error: "PDF service unavailable. Please try again later." }`
- **Filename:** Sanitize `hatcheryName` — replace `/[^a-zA-Z0-9_-]/g` with `_` to avoid invalid filename characters.

### 2c. Install Playwright

Run `npm install playwright` in `aquaspec/`. After install, run `npx playwright install chromium` to download the Chromium binary.

## Slice 3: Frontend — Preview Page + Modal Updates

### 3a. `src/app/proposal/preview/page.tsx`

**Type:** Client component (`"use client"`)

**Behavior:**
1. On mount, read `recommendation` and `budgetaryEstimateEnabled` from Zustand
2. If `recommendation` is null, redirect to `/` (or show error "No sizing data available. Please complete the wizard first.")
3. POST to `/api/proposal/preview` with `{ recommendation, includeBudgetary: budgetaryEstimateEnabled }`
4. Show a loading spinner while the HTML is being fetched
5. Render the HTML via `dangerouslySetInnerHTML` in a full-page container
6. Show two buttons in a sticky bottom bar:
   - **"← Back to Wizard"** — navigates to `/`
   - **"Download PDF"** — triggers PDF download

**PDF download flow:**
1. Click "Download PDF" → show loading state on button ("Generating...")
2. POST to `/api/proposal/pdf` with `{ recommendation, includeBudgetary: budgetaryEstimateEnabled }`
3. On success: create a blob from the response, trigger browser download via `URL.createObjectURL` + `<a>` click
4. On timeout/error: show toast with the error message from the API

**Styling:**
- Full-width container with white background
- Sticky bottom bar with border-top
- Back button (outline variant) on left, Download button (primary, filled) on right
- Loading state: centered spinner with "Preparing proposal..."

## Acceptance Criteria

### AC-1: Template renders correctly
**Given** a valid `HatcheryRecommendation`  
**When** the API route `POST /api/proposal/preview` is called  
**Then** it returns HTML containing: the Lotus Ozone logo, hatchery name, formatted date, per-system input parameters table, per-system engineering results table, and footer text. No React component imports are used in the HTML generation path.

### AC-2: Aggregate summary only in multi-system mode
**Given** a recommendation with `mode: "aggregate"`  
**When** the preview HTML is rendered  
**Then** no aggregate summary section appears.  
**Given** a recommendation with `mode: "multi_system"` and `aggregateSummary` present  
**When** the preview HTML is rendered  
**Then** an aggregate summary table appears showing total flow, total ozone, total oxygen, and combined equipment list.

### AC-3: PDF is generated and downloadable
**Given** a valid recommendation  
**When** `POST /api/proposal/pdf` is called  
**Then** the response is `application/pdf` with a `Content-Disposition: attachment` header, the filename matches `LotusOzone_Proposal_{Name}_{Date}.pdf`, and the PDF buffer is non-empty (> 1000 bytes).

### AC-4: Budgetary estimate toggle works
**Given** the proposal modal is open  
**When** the user checks "Include Budgetary Estimate" and clicks Download  
**Then** the preview page includes the budgetary estimate section.  
**Given** the proposal modal is open  
**When** the user leaves the checkbox unchecked and clicks Download  
**Then** the preview page does NOT include the budgetary estimate section.

### AC-5: Preview page renders from Zustand data
**Given** the user has completed the wizard with a valid recommendation in Zustand  
**When** the user navigates to `/proposal/preview`  
**Then** the page fetches the HTML from the preview API, displays it full-page, and shows Back + Download buttons in a sticky footer bar.

### AC-6: Preview page handles missing data
**Given** no recommendation exists in Zustand (null)  
**When** the user navigates to `/proposal/preview`  
**Then** an error message is shown: "No sizing data available. Please complete the wizard first." and the user is not stuck.

### AC-7: PDF timeout handled
**Given** Playwright takes longer than 15 seconds  
**When** the PDF API is called  
**Then** it returns 504 with `{ error: "PDF generation timed out. Please try again." }` and does not crash the server.

### AC-8: Chromium unavailable handled
**Given** Chromium binary is missing or crashes  
**When** the PDF API is called  
**Then** it retries once, and if still failing, returns 503 with `{ error: "PDF service unavailable. Please try again later." }`.

### AC-9: Download button navigates to preview
**Given** the ProposalPreview modal is open  
**When** the "Download PDF" button is clicked  
**Then** the modal closes and the browser navigates to `/proposal/preview`.

### AC-10: Existing tests still pass
**Given** the PDF pipeline code is added  
**When** `npm test -- --run` is executed  
**Then** all 11 existing engine tests pass with no changes needed.

### AC-11: Build succeeds
**Given** all changes are written  
**When** `npm run build` is executed  
**Then** the Next.js build completes with zero TypeScript errors.

### AC-12: Protected files untouched
**Given** the implementation is complete  
**When** `git diff HEAD -- src/lib/sizing-engine/ src/app/api/size/ src/app/api/defaults/ src/components/wizard/ src/components/results/ src/data/sizing-rules.json` is run

## Constraints

1. **Zero React imports in `proposal-html.ts`** — this is a pure string-generating module.
2. **No client-side exposure of `sizing-rules.json`** — the PDF API runs server-side only.
3. **No modifications to the sizing engine** or its tests.
4. **No changes to existing API routes** (`/api/size`, `/api/defaults`).
5. **A4 format only** for PDF output.
6. **Filename must match** `LotusOzone_Proposal_{sanitizedHatcheryName}_{YYYY-MM-DD}.pdf`.
7. **Budgetary estimate toggle defaults to OFF** (false).

## Non-Goals

- Print button or browser print dialog integration
- Email proposal directly from the app
- PDF caching or storage
- Multi-language PDF support
- Customizable templates (single template only)
- PDF generation for individual systems (always the full proposal)
- Save/load of hatchery configurations (deferred to next slice)
- Admin portal (deferred entirely)

## Known Unknowns

- Playwright Chromium binary download may fail on restricted networks — the install step should be verified after `npx playwright install chromium`.
- The `dangerouslySetInnerHTML` approach on the preview page means the HTML is trusted (it's server-rendered). No user input enters the HTML generation path, so XSS risk is minimal, but verify the sanitization.
- The logo URL resolution: `req.nextUrl.origin` works for the preview API route. For the PDF API route, the HTML is rendered server-side — the logo URL needs to be an absolute URL that Chromium can load. Use `http://localhost:3000/LotusOzoneLogo.png` in dev, and the Vercel deployment URL in production. Solve this by passing the base URL as a parameter or using an environment variable.
